# 核心指令：针对性优化简历内容

请根据提供的**职位描述 (JD)**，严格修改我的简历 JSON，以最大化通过 **ATS (Applicant Tracking System)** 的几率。

## 目标与方法论

1.  **关键词优化：** 在不违背我真实背景的前提下，将 JD 中的**核心关键词**自然地融入到 `experience`, `projects`, 和 `skills` 部分。
2.  **内容提炼方法论：** 所有 **bullet point** 的内容**必须**基于 **XYZ 方法论 (Accomplished [X] as measured by [Y] by doing [Z])** 进行重写或精炼。
    * **X (Achieved/Result):** 你取得了什么成就/结果。
    * **Y (Metric/Quantification):** 用数字、百分比等量化你的成就。
    * **Z (Action/Context):** 你采取了什么具体行动来实现它。

## 严格约束与格式要求（重点关注篇幅控制）

1.  **真实性底线：** 简历的修改**必须基于我现有的背景和经历**。**严禁**添加任何虚假的、我未曾参与过的经历、项目。

2.  **项目替换机制（Projects）：**
    * 我提供了一个额外的列表：`extraProjects`，这些是我做过但未列在当前简历中的项目。
    * LLM 必须根据 JD 的要求，**从当前的 `projects` 或 `extraProjects` 中选择最相关的项目，并用它们来替换当前 `projects` 部分的内容。**

3.  **篇幅和精简要求（最重要）：**
    * **我的 JSON 简历内容已满，渲染后恰好一页。**
    * **核心原则：以“替换”而非“增减”为主，以确保篇幅稳定。**
    * 在优化 `experience` 和 `skills` 时，**严禁**单纯增加新的 bullet point 或大段文字。**如果增加一行内容，必须删除至少一行等量的旧内容。**
    * **项目替换的篇幅控制：** 如果替换 `projects` 中的内容，**替换后的项目内容（包括标题、描述、要点总长度）必须与被替换的原内容保持大致相同的篇幅。** （例如：替换一个有5个要点的项目，新替换的项目也应有大约5个要点或等量篇幅。）
    * **最终生成的 JSON 转换成 PDF 后，必须严格控制在单页篇幅内。**

4.  **JSON 格式要求：**
    * **输出必须严格遵循我提供的原始 JSON 格式。**
    * **允许修改**的部分：`experience`, `projects`, `skills`。
    * **严禁修改**的部分：`basics`, `education`。
